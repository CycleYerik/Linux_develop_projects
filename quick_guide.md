# 说明

本文档用于 Ubuntu 笔记本上的 Linux 基础开发学习，重点是通过可运行 demo 建立工程能力。

建议学习节奏：

- 先完成最小可运行版本（MVP），保证能编译、能运行、能看到结果。
- 再逐步加入稳定性和可维护性能力：日志、错误处理、配置文件、单元测试。
- 每个项目都做一次复盘：输入输出是什么、边界条件是什么、哪里容易出错。

# 分类

## 通用开发基础（建议先完成）

### 项目0：开发环境与工程模板

- 目标：建立可复用的 C/C++ Linux 项目模板。
- 关键点：`gcc/g++`、`make`、`cmake`、`gdb`、`valgrind`、`strace`。
- 建议内容：
  - 建一个标准目录：`src/`、`include/`、`tests/`、`scripts/`、`build/`。
  - 写 `CMakeLists.txt`，支持 Debug/Release 两种构建。
  - 增加脚本：一键编译、一键运行、一键清理。
- 进阶扩展：
  - 接入 `clang-format` 和 `clang-tidy`。
  - 增加 Git pre-commit（格式检查 + 基础静态检查）。

### 项目1：日志与配置系统（给后续所有 demo 复用）

- 目标：写一个轻量公共模块，供所有 demo 直接复用。
- 关键点：日志分级、时间戳、文件轮转、配置读取（ini/json/yaml 任一）。
- 建议内容：
  - `log_info/log_warn/log_error` 三个级别。
  - 支持终端输出 + 文件输出。
  - 支持从配置文件读取端口号、线程数、采样周期。
- 进阶扩展：
  - 日志异步写盘（生产者-消费者模型）。
  - 支持热加载配置（`SIGHUP`）。

## Linux应用开发

### 多线程-多传感器处理程序

- 场景：模拟温度/湿度/光照/压力等多个传感器并发采集。
- 目标：
  - 每个传感器一个采集线程（或线程池任务）。
  - 主线程汇总数据并定时打印统计结果。
- 关键技术点：
  - `pthread_create`、互斥锁、条件变量、读写锁。
  - 环形缓冲区（ring buffer）与线程安全队列。
- MVP建议：
  - 每个线程每 200ms 产生一次随机数据。
  - 汇总线程每 1s 输出最大值/最小值/平均值。
- 进阶扩展：
  - 增加阈值告警机制（超限打印报警）。
  - 增加 CSV 数据落盘，方便后续可视化。

### 多进程与进程通信-传感器数据传输

- 场景：采集进程与分析进程分离，通过 IPC 通信。
- 目标：
  - 采集进程只负责采集。
  - 分析进程只负责清洗/统计/告警。
- 关键技术点：
  - 管道（pipe/FIFO）、消息队列、共享内存、信号量。
  - 进程退出与资源回收（`wait`/`waitpid`）。
- MVP建议：
  - 先用匿名管道实现一对一通信。
  - 再替换为共享内存 + 信号量，比较性能差异。
- 进阶扩展：
  - 增加统一数据协议（消息头 + 长度 + 校验）。
  - 加入异常恢复：消费者重启后可继续处理。

### 多进程管理-监督进程和业务进程

- 场景：实现一个简版 supervisor，守护多个业务子进程。
- 目标：
  - 监督进程可拉起、监控、重启子进程。
  - 记录重启次数与异常原因。
- 关键技术点：
  - `fork/exec`、`waitpid`、信号处理（`SIGCHLD`、`SIGTERM`）。
  - 进程状态机（运行中/退出/重启中/熔断）。
- MVP建议：
  - 配置文件里定义 2~3 个子进程命令。
  - 子进程异常退出后自动拉起，超过阈值进入熔断。
- 进阶扩展：
  - 增加心跳机制（unix socket 或共享内存）。
  - 实现简单命令接口：`status/start/stop/restart`。

### 网络-基础TCP通信模拟

- 场景：模拟“客户端上传传感器数据，服务端接收并回包”。
- 目标：
  - 完成 TCP 服务端和多个客户端。
  - 支持并发连接与基本协议解析。
- 关键技术点：
  - `socket/bind/listen/accept/connect/send/recv`。
  - 阻塞/非阻塞、`select/poll/epoll` 对比。
- MVP建议：
  - 文本协议：`sensor_id,timestamp,value`。
  - 服务端回包：`OK` 或错误码。
- 进阶扩展：
  - 改为定长包头 + 变长包体协议。
  - 加入重连机制、超时控制、心跳保活。

### 虚拟串口控制程序

- 场景：不接真实硬件，用伪串口模拟 MCU 与上位机通信。
- 目标：
  - 使用虚拟串口工具（如 `socat`）建立串口对。
  - 实现主机端串口读写程序。
- 关键技术点：
  - `termios` 配置（波特率、校验位、停止位、超时）。
  - 帧格式设计、粘包拆包、CRC 校验。
- MVP建议：
  - 一个进程模拟“设备端”，一个进程模拟“控制端”。
  - 支持 3 条命令：读状态、写参数、心跳。
- 进阶扩展：
  - 增加命令重试与 ACK/NACK 机制。
  - 增加串口通信日志回放工具。

### 新增：事件驱动IO模型对比（select/poll/epoll）

- 目标：通过同一业务逻辑，对比 3 种 IO 复用模型。
- 建议内容：
  - 写一个 echo server，分别实现 `select`、`poll`、`epoll` 三版。
  - 用压测脚本比较连接数、CPU 占用和响应延迟。
- 输出结果：
  - 形成一张对比表（实现复杂度、性能、适用场景）。

### 新增：定时器与任务调度器

- 目标：实现一个简易任务调度器（周期任务 + 一次性任务）。
- 关键点：`timerfd` 或最小堆时间轮思想、任务取消、任务漂移处理。
- 建议内容：
  - 支持每秒采集任务、每 5 秒统计任务、每天轮转日志任务（模拟）。
  - 将调度器集成到传感器项目中替代 `sleep` 粗暴轮询。

## Linux驱动开发

### 前置准备（在进入驱动前）

- 熟悉最小模块：`module_init`、`module_exit`、`insmod/rmmod/lsmod/dmesg`。
- 学会写用户态测试程序与内核态日志联调。
- 确认当前 Ubuntu 内核头文件和编译环境可用。

### 虚拟字符设备驱动

- 场景：实现一个不依赖真实硬件的字符设备驱动。
- 目标：
  - 创建设备节点，支持 `open/read/write/ioctl`。
  - 用户态程序可读写驱动中的缓冲区。
- 关键技术点：
  - `cdev` 注册、主次设备号申请、`file_operations`。
  - `copy_to_user/copy_from_user`、并发访问保护。
- MVP建议：
  - 内核模块维护一个固定大小缓冲区。
  - 用户程序支持写入字符串并读回验证。
- 进阶扩展：
  - 支持阻塞/非阻塞读写与 `poll`。
  - 用 `ioctl` 设置设备工作模式（如回显、只读、统计）。

### 新增：Procfs/Sysfs 交互驱动

- 目标：学习驱动参数暴露与运行时观测。
- 建议内容：
  - 在 `/proc` 或 `/sys` 暴露计数器、状态位、配置项。
  - 用户态通过 `cat/echo` 查看和修改参数。
- 进阶扩展：
  - 与字符设备结合，形成“控制面 + 数据面”结构。

## 每个项目统一验收标准

- 能一键构建：`cmake && make` 或 `make`
- 能一键运行：有启动脚本和参数示例
- 有最少 3 类日志：启动、异常、退出
- 有 README：说明功能、运行方式、目录结构
- 有基础测试：至少覆盖正常路径 + 一个异常路径

## 两个月学习总计划（非全日制）

### 时间预算建议

- 总周期：8 周。
- 每周投入：8~12 小时（工作日每天 1~1.5 小时 + 周末半天）。
- 总投入：约 64~96 小时。
- 时间分配比例：
  - 编码实践：60%
  - 调试与排错：25%
  - 文档复盘：15%

### 学习策略（确保能坚持）

- 每周只做 1 个“主项目” + 1 个“小收尾”。
- 每个主项目都遵循三步：
  1. 跑通最小版本（30% 时间）
  2. 做稳定性和异常处理（50% 时间）
  3. 写 README 与复盘（20% 时间）
- 若某周超时，优先保留：可运行 + 可复现；进阶功能可延期。

## 8周详细执行计划

### 第1周：环境、模板、调试链路打通

- 本周目标：
  - 建立统一工程模板和脚本。
  - 确保 `gdb/valgrind/strace` 都能实际用起来。
- 必做任务：
  - 创建项目骨架：`src/ include/ tests/ scripts/ build/ docs/`。
  - 完成 `CMakeLists.txt`（Debug/Release）。
  - 写 `scripts/build.sh`、`scripts/run.sh`、`scripts/clean.sh`。
  - 写一个 `hello_demo` 并用 `gdb` 断点、`valgrind` 查内存、`strace` 看系统调用。
- 本周产出：
  - 一个可复用模板仓库（后续 demo 直接复制）。
  - `docs/dev_setup.md`（记录工具安装和常见问题）。

### 第2周：日志与配置模块（后续全复用）

- 本周目标：实现可复用基础模块，降低后续项目重复劳动。
- 必做任务：
  - 日志模块：等级、时间戳、终端+文件输出。
  - 配置模块：读取 `ini` 或 `json`（二选一）。
  - 错误码规范：统一返回值与错误打印格式。
- 本周产出：
  - `libcommon.a` 或 `common/` 源码目录。
  - `examples/log_and_config_demo`。
  - README：如何在其他项目中引入。

### 第3周：多线程多传感器采集

- 本周目标：完成并发采集 + 汇总统计闭环。
- 必做任务：
  - 4 个模拟传感器线程（温湿度光照压力）。
  - 线程安全队列或环形缓冲区。
  - 汇总线程每秒输出统计值。
- 本周产出：
  - `apps/thread_sensor_demo/`。
  - 可配置参数：采样间隔、线程数量、告警阈值。
  - 一份并发问题记录（如锁竞争、死锁规避方案）。

### 第4周：多进程与IPC

- 本周目标：实现“采集进程 -> 分析进程”的数据链路。
- 必做任务：
  - 版本A：匿名管道通信。
  - 版本B：共享内存 + 信号量通信。
  - 对比两版本吞吐和延迟（粗粒度即可）。
- 本周产出：
  - `apps/ipc_sensor_demo/pipe_version`。
  - `apps/ipc_sensor_demo/shm_version`。
  - `docs/ipc_compare.md`（优缺点对照）。

### 第5周：监督进程与业务进程管理

- 本周目标：做一个简版 supervisor。
- 必做任务：
  - 监督进程拉起 2~3 个子进程。
  - 异常退出自动重启，带重启上限与熔断。
  - 处理退出信号（Ctrl+C 时优雅停止）。
- 本周产出：
  - `apps/supervisor_demo/`。
  - `supervisor.conf` 示例配置。
  - `docs/process_lifecycle.md`（状态流转图）。

### 第6周：TCP通信 + IO复用模型

- 本周目标：网络通信可用，并理解 `select/poll/epoll`。
- 必做任务：
  - TCP server/client 跑通。
  - 完成 `select` 版本服务器。
  - 增加 `epoll` 版本（`poll` 可选做）。
- 本周产出：
  - `apps/tcp_demo/`。
  - 压测脚本（可用 `ab`、`wrk` 或自写客户端循环连接）。
  - `docs/io_multiplexing_notes.md`（场景建议）。

### 第7周：虚拟串口 + 定时任务调度

- 本周目标：接近嵌入式真实场景的数据交互与调度。
- 必做任务：
  - 用 `socat` 建立伪串口对并完成读写。
  - 协议实现：命令、应答、校验（可简化 CRC）。
  - 用 `timerfd` 做周期任务调度。
- 本周产出：
  - `apps/virtual_uart_demo/`。
  - `apps/timer_scheduler_demo/`。
  - `docs/serial_protocol.md`（帧结构说明）。

### 第8周：字符设备驱动入门 + 总整合

- 本周目标：完成一个可读写的虚拟字符设备，并做两个月总结。
- 必做任务：
  - 驱动模块：`open/read/write/ioctl`。
  - 用户态测试程序读写验证。
  - 可选：增加 `procfs/sysfs` 状态项。
- 本周产出：
  - `driver/virtual_char_dev/`。
  - `apps/driver_test/`。
  - `docs/final_review.md`（收获、薄弱点、下一阶段计划）。

  - 30 分钟读资料 + 45 分钟编码 + 15 分钟记录问题

  - 这周卡住在哪里？根因是什么？

## 风险控制（非全日制最关键）

- 如果时间不够，优先级按下列顺序保留：
  1. 主功能可运行
  2. 日志和错误处理
  3. 文档与测试
  4. 性能优化和高级特性
- 每周至少保留 1 个“看得见成果”的可执行程序，避免挫败感。
- 每两周做一次“整合周”：把前面模块连起来，而不是只做零散 demo。
